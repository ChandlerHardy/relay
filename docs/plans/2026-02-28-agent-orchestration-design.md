# P3: Agent Orchestration — Design

> **For Claude:** REQUIRED SUB-SKILL: Use superpowers:executing-plans to implement the corresponding implementation plan task-by-task.

**Goal:** Turn Relay from a manual session launcher into an autonomous agent staff that takes high-level goals and executes them end-to-end.

**Architecture:** Event-driven API orchestrator. Relay server stays as infrastructure (spawn, stream, store). A new orchestration engine makes Anthropic API calls at key trigger points (plan, evaluate, summarize) to drive assignment progress. Claude Code sessions handle execution; API calls handle decisions.

---

## Core Concepts

### Assignment

A high-level goal submitted by the user. The PM breaks it into tasks, each task becomes one or more Claude Code sessions.

```
Assignment {
  id: string
  goal: string                          // user's high-level input
  projectDir: string                    // target project
  status: "planning" | "active" | "completed" | "failed"
  autonomyLevel: "autonomous" | "available"
  plan: TaskPlan | null                 // generated by PM's planning call
  decisions: Decision[]                 // pending/resolved decision log
  createdAt: string
  completedAt: string | null
  summary: string | null                // PM's completion report
}

TaskPlan {
  tasks: Task[]
}

Task {
  id: string
  description: string
  role: "research" | "implement" | "review"
  status: "pending" | "active" | "completed" | "failed" | "retrying"
  dependencies: string[]                // task IDs that must complete first
  sessionIds: string[]                  // Claude Code sessions spawned for this task
  evaluation: string | null             // PM's verdict on completion
  retryCount: number
}

Decision {
  id: string
  question: string                      // what the PM needs decided
  options: string[]                     // suggested choices
  context: string                       // why this decision matters
  status: "pending" | "resolved"
  resolution: string | null             // user's answer or PM's autonomous choice
  resolvedBy: "user" | "pm"
  createdAt: string
  resolvedAt: string | null
}
```

### Session Extensions

Existing sessions gain optional links back to assignments:

```
Session (existing fields) + {
  assignmentId?: string
  taskId?: string
  role?: "research" | "implement" | "review"
}
```

Standalone sessions (no assignment) continue to work exactly as before.

## Orchestration Engine

New module: `orchestrator.ts`

### Trigger Points

**1. Plan (assignment created)**
- Input: goal, project context (CLAUDE.md, codebase structure), available skills
- API call: PM system prompt + user goal
- Output: structured task breakdown with dependencies and roles
- Action: store plan, mark first eligible tasks as active, spawn sessions

**2. Evaluate (session completed)**
- Input: task description, session output, assignment context
- API call: PM evaluates whether task succeeded
- Output: pass/fail/retry verdict + reasoning + next actions
- Actions:
  - Pass → mark task complete, check dependency graph, spawn next eligible tasks
  - Fail → retry (up to 2x) with adjusted prompt, or surface decision to user
  - Ambiguous → if `available`, queue decision; if `autonomous`, make best judgment

**3. Summarize (all tasks complete)**
- Input: full assignment context, all task evaluations
- API call: PM generates completion report
- Output: summary of what was accomplished, any caveats
- Action: mark assignment complete, notify user

### PM System Prompt

The PM persona understands:
- Claude Code's skill ecosystem (TDD, ralph-local, code-review-local, etc.)
- How to craft prompts that instruct sessions to use appropriate internal workflows
- Task dependency ordering and parallelism
- When to escalate vs. decide autonomously

Role-specific prompt templates:
- **Research tasks:** "Investigate X. Report findings as structured markdown. Do not make code changes."
- **Implementation tasks:** "Implement X using TDD. Use the /commit skill when done. Run tests until green before committing."
- **Review tasks:** "Review the changes in the last N commits. Report issues found. Do not make changes unless critical."

### API Integration

- Uses Anthropic API (via z.ai proxy or direct `@anthropic-ai/sdk`)
- Model: claude-sonnet-4-6 for orchestration calls (fast, cheap, smart enough for planning/evaluation)
- Structured output via tool_use for reliable JSON parsing of plans and evaluations
- Estimated cost per assignment: ~$0.05-0.20 for orchestration calls (the Claude Code sessions are the real cost)

## Autonomy Dial

Each assignment has an `autonomyLevel`:

- **`autonomous`** — PM makes all decisions, only notifies on completion or catastrophic failure. Logs reasoning for every decision made.
- **`available`** — PM surfaces ambiguous decisions to the user. Buffers decisions when user is offline. If a decision blocks progress and user doesn't respond within a configurable timeout, PM makes its best judgment and logs it.

User can toggle this per-assignment or globally, even mid-flight. The toggle is visible in the assignment view.

Future (P4): When Telegram integration is live, `available` mode routes decisions to Telegram via ZeroClaw.

## Two-Layer Quality Model

### Micro Level (within each session)
Claude Code's own skill ecosystem handles internal quality:
- TDD / test-driven-development for implementation
- ralph-local for iterative goal-driven loops
- Code review skills for self-review before completion
- The PM crafts prompts that instruct sessions to use these workflows

### Macro Level (between sessions)
The PM orchestration engine handles assignment-level quality:
- Evaluates session output against task description
- Detects when a task went off-track
- Retries with adjusted prompts
- Ensures dependency ordering is respected

The PM doesn't need to babysit mid-session because sessions are self-supervised.

## API Endpoints (New)

| Endpoint | Method | Purpose |
|----------|--------|---------|
| `/api/assignments` | GET | List all assignments |
| `/api/assignments` | POST | Create assignment (goal + projectDir + autonomyLevel) |
| `/api/assignments/:id` | GET | Get assignment details (includes tasks, decisions) |
| `/api/assignments/:id` | DELETE | Cancel assignment (kills running sessions) |
| `/api/assignments/:id/autonomy` | PATCH | Toggle autonomy level |
| `/api/assignments/:id/decisions/:did` | POST | Resolve a pending decision |

Existing session endpoints unchanged. Sessions created by the orchestrator use the existing `POST /api/sessions` internally.

- Simpler than current session dialog: just goal text + project directory + autonomy level
- No prompt field (PM generates the prompts)

## Persistence

- Assignments stored in `~/.relay/assignments.json` alongside existing `sessions.json`
- Same pattern: save on state changes, load on startup
- Assignment references sessions by ID (sessions already persisted separately)
